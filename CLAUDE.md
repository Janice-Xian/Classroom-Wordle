# Classroom Wordle — Claude 协作规则

## 项目简介

课堂英语单词复习投影游戏。英语老师在课堂上打开网页，从已保存的词库中抽取 5 个单词，全班学生看着投影仪屏幕口头猜测，老师代为输入。系统通过颜色反馈（绿/黄/灰）引导学生推理出目标单词。

## 技术栈

- **前端框架**：React 18（函数式组件 + Hooks）
- **样式**：Tailwind CSS 3
- **构建工具**：Vite 5
- **后端存储**：Supabase（PostgreSQL + RLS）
- **游戏状态**：React useState / sessionStorage

## 改代码前必读文档

每次修改代码前，必须先读以下三份文档，理解项目背景和当前状态：

1. `PRD.md` — 产品需求文档，所有功能的权威来源
2. `ARCH.md` — 架构说明，目录结构、数据表、状态管理
3. `project_state.md` — 当前项目状态、已知问题、下一步计划

## 开发规矩

### 只改指定文件
- 只修改用户明确指定的文件
- 不要因为"顺手"而修改其他文件
- 不要跨组件重构代码结构

### 不添加 PRD 之外的功能
- 严格按 `PRD.md` 实现，不主动添加新功能
- PRD 明确排除的功能（色盲模式、积分系统、文件上传等）不做
- 如果发现需求模糊，先询问，不要自行决策

### 界面文字用中文
- 所有面向用户的文字（按钮、提示、弹窗）使用中文
- 组件内部注释可用中英文
- 变量名、函数名用英文

### 单个文件不超过 300 行
- 超过 300 行时提醒用户考虑拆分，不要擅自拆分
- 拆分前需用户确认

### 其他
- 改动前先读文件，不要凭假设修改
- 保持现有代码风格，不要引入新的库或工具
- 不要删除或重命名现有函数，除非用户明确要求

## 每次完成任务后的固定流程

每次完成用户交给的修改任务后，必须自动执行以下三步，不需要用户提醒：

### 第一步：更新 project_state.md
- 将本次完成的内容移入"已完成"列表
- 如果发现新问题，加入"已知问题"列表
- 更新"下一步计划"（移除已完成项，调整优先级）

### 第二步：git commit
```
git add .
git commit -m "简洁描述这次改了什么（中文）"
```
commit 信息用中文，简洁描述本次变更内容。

### 第三步：git push
```
git push
```
如果 push 失败，告知用户错误原因并给出解决方法，不要静默失败。
